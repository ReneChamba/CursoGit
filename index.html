Identidad

Esto es importante porque los "commits" de Git usan esta información, y es introducida de manera inmutable en los commits que envías:

Configurar identidad de forma global

pg-21

git config --global user.name "Ronny": Agregar un nombre global para nuestro usuario.

git config --global user.email ronn@gmail.com: Agregar un email para nuestro usuario.

En sistemas Windows, Git busca el archivo .gitconfig en el directorio $HOME (para mucha gente será (C:\Users\$USER).

Configurar identidad para un proyecto especifico

pg-21

Simplemente omitimos --global, esto configurara un nombre y email exclusivo para el repositorio actual.

git config user.name Ronny : Agregar un nombre para nuestro repositorio actual.

git config user.email ronn@gmail.com: Agregar un email para nuestro repositorio actual.

Estos datos se agregaran en el archivo config de nuestro repositorio

Clonar un repositorio

Clonar un repositorio con nombre especifico

Guardando cambios en el Repositorio

El siguiente paso es realizar algunos cambios y confirmar instantáneas de esos cambios en el repositorio cada vez que el proyecto alcance un estado que quieras conservar.

Estados de los archivos

pg:25

Recuerda que cada archivo de tu repositorio puede tener dos estados: rastreados y sin rastrear.

tracked files: son todos aquellos archivos que estaban en la última instantánea del proyecto, pueden ser archivos sin modificar, modificados o preparados.

Untracked files: cualquier otro archivo en tu directorio de trabajo que no estaba en tu última instantánea y que no está en el área de preparación.

Ver Estados de los archivos

pg:26

git status : La herramienta principal para determinar qué archivos están en qué estado.

Rastrear archivos nuevos

pg:27

git add nameArchivo:Git comenzara a darle seguimiento al archivo.

Preparar Archivos Modificados

pg:28

Una vez git este dando seguimiento a un archivo, Vamos a modificar un archivo que esté rastreado.

git add archivo: Agregamos el archivo modificado al area de preparación

El archivo ahora esta preparado para la proxima confirrmación

Dos estados de un archivo

Supongamos que tenemos dos archivos (index.html y style.css) listos para confirmar, pero antes de la confirmacion (commit), recordamos que tenemos que realizar un cambio en index.html. Realizamos el cambio en index.html, y sí vemos de nuevo el estado de nuestros archivos (git status), podemos apreciar algo raro.

Podremos ver que el archivo index.html tiene dos estados, un estado listo para confirmar y otro estado modificado, entonces:

Si confirmas ahora, se confirmará la versión de index.html que tenías la última vez que ejecutaste git add y no la versión que ves ahora en tu directorio de trabajo, al ejecutar git status. Si modificas un archivo luego de ejecutar git add, deberás ejecutar git add de nuevo para preparar la última versión del archivo:

Sacar un archivo de staging area

git restore --staged nameArchivo:En caso que el arhivos no este en el staging area, NO causará error.

Ignorar archivos

pg:30

A veces, tendrás algún tipo de archivo que no quieres que Git añada automáticamente o más aun, que ni siquiera quieras que aparezca como no rastreado.

Crear un archivo .gitignore, por lo general se lo crea en el directorio raiz del proyecto ,dentro de él especifcar los archivos, patros, rutas a directorios que desea qu Git ignore.

Crear archivo: touch .gitignore

Configurar archivo .gitignore

  • *: se utiliza como una coincidencia comodín.
  • /: se usa para ignorar las rutas relativas al archivo .gitignore.
  • #: utilizado para escribir un comentario

Ejemplos de reglas para ignorar archivos

  • # Ignorar archivo miArchivo.txt: miArchivo.txt
  • # Ignorar todos los archivos con extensión .txt : *.txt
  • # Ignorar todos los archivos del directorio sass : sass/
  • # ignorar doc/notes.txt, pero no este: doc/server/arch.txt : doc/*.txt
  • # Ignorar un arhivo desendiente directo de la raiz : /ejemplo.txt Si existiera otro archivo ejemplo.txt dentro de un subdirectorio, no lo tomara en cuenta ya que no es descendiente directo de la raiz.

Nota Importante: Para que tomen efecto y Git ignore los archivos especificados en el archivo .gitignore se debe primero hacer un git add . y luego un git commit -m "mensaje"

Ver los Cambios Preparados y No Preparados

pg:31

Si el comando git status es muy impreciso para ti - quieres ver exactamente que ha cambiado, no solo cuáles archivos lo han hecho - puedes usar el comando git diff .

Ver Qué has cambiado pero aun no has preparado?

git diff : Este comando compara lo que tienes en tu directorio de trabajo con lo que está en el área de preparación. El resultado te indica los cambios que has hecho pero que aun no has preparado

git diff solo mostrara el archivo con las modificaciones que no hay sido preparadas aun, caso contrario no mostrara nada.

¿ Ver qué has preparado y está listo para confirmar?

git diff --staged[--cached :son sinonimos ] :ver lo que has preparado y será incluido en la próxima confirmación.Este comando compara tus cambios preparados con la última instantánea confirmada.

git diff --staged mostrara lo que has preparado hasta ahora, caso contrario no mostrará nada.

Confirmar tus Cambios

pg: 34

Recuerda que cualquier cosa que no esté preparada - cualquier archivo que hayas creado o modificado y que no hayas agregado con git add desde su edición - no será confirmado. Se mantendrán como archivos modificados en tu disco.

git commit -m "descripcion"

Recuerda que la confirmación guarda una instantánea de tu área de preparación. Todo lo que no hayas preparado sigue allí modificado;

Saltar el Área de Preparación

pg: 35

Si quieres saltarte el área de preparación, Git te ofrece un atajo sencillo. Añadiendo la opción -a al comando git commit harás que Git prepare automáticamente todos los archivos rastreados antes de confirmarlos, ahorrándote el paso de git add.

git commit -am "mensaje": Fíjate que en este caso no fue necesario ejecutar git add sobre el archivo ejemplo.html antes de confirmar.

Nota:

Este combinación de comandos solo sera valido para los archivos qu tenga seguimiento.

Eliminar Archivos

pg:36

Caso 1: Eliminar el seguimiento de archivos que sólo están en staging area (no se ha hecho ningun commit aun )

git reset HEAD archivo: Con esto git dejara de dar seguimiento al archivo.

git reset HEAD: Con esto git dejara de dar seguimiento a todos los archivos de la rama actual.

Caso 2: Eliminar del repositorio archivos que ya se han confirmado (se ha hecho commit)

Una vez confirmados los cambios, es decir, una vez has hecho commit a esos archivos por primera vez, los archivos ya forman parte del repositorio y sacarlos de allí requiere algún paso adicional. Pero es posible gracias a la instrucción rm, opciones:

git rm arhivo: Cuidado este comando elimina el arhivo de repositorio Git pero además lo elimina de nuestro Directorio de Trabajo, osea de nuestro proyecto.

git rm --cached archivo : Elimina el archivo de repositorio pero el archivo aún permanecera en nuestro Directorio de Trabajo, osea aún estara en nuestro proyecto.

git rm -r --cached nombreDirectorio : Borrar un directorio y todos sus contenidos, eso archivos aun permaneceran en nuestro Directorio de Trabajo, osea aún estara en nuestro proyecto.

Luego de eliminar los archivos del repositorio de Git,estos se marcaran como archivos sin seguimiento (Untracked), posterior tendras que asegurarte de que sean ignorados en el archivo .gitignore para que deje de aparecer como archivo sin seguimiento. y para que no los volvamos a meter en el repositorio cuando confirmemos cambios más adelante con commit, Luego hacer commit para confirmas los cambios

Ver el Historial de Confirmaciones

La herramienta más básica y potente para hacer esto es el comando git log. Nos permite ver el historial de confirmaciones commit , probablemente quieras mirar atrás para ver qué modificaciones se han llevado a cabo

Ejecutar Comando

pg:38

git log : lista las confirmaciones hechas sobre ese repositorio en orden cronológico inverso. Es decir, las confirmaciones más recientes se muestran al principio.

Opciones para ejecutar log

pg:39

El comando git log proporciona gran cantidad de opciones para mostrarte exactamente lo que buscas. Aquí veremos algunas de las más usadas.

  • Limitar las salidas de confirmaciones (-numero)

    git log -2 :Mostrara las dos ultimas confirmaciones

  • Mostrar las diferencias introducidas en cada confirmación (-p)

    git log -p: Mostrara las confirmaciones agregando las diferencias en cada confirmacion

  • Combinacion de las dos opciones anteriores (-p -2)

    git log -p -2: Listara las dos ultimas confirmaciones agregando las diferencias en cada confirmacion

  • El comando git log puede usar una serie de opciones de resumenes. Veremos algnos de ellos

    • git log --stat: la opcion --stat permite ver algunas estadísticas de cada confirmación,

    • git log --stat -1: podemos combinar con las opciones vista anteriormente.

    • git log -p --stat -1: podemos combinar con las opciones vista anteriormente.

    • git log --oneline: Listar cada confirmacion en una sola

    • git log --oneline -2: Listar las dos ultimas confirmaciones en una sola linea.

  • Hay muchas opciones máss que permiten personalizar y combinar la salida con el comando git log, revisar desde la pagina 38 hasta 45

Deshacer Cosas

Revisaremos algunas herramientas básicas usadas para deshacer cambios que hayas hecho. Ten cuidado, a veces no es posible recuperar algo luego que lo has deshecho. Esta es una de las pocas áreas en las que Git puede perder parte de tu trabajo si cometes un error.

Deshacer una confirmación

pg:45

git commit --amend:Si quieres rehacer la confirmación, puedes reconfirmar con la opción --amend

Este comando utiliza tu área de preparación para la confirmación. Si no has hecho cambios desde tu última confirmación (por ejemplo, ejecutas este comando justo después de tu confirmación anterior), entonces la instantánea lucirá exactamente igual y lo único que cambiarás será el mensaje de confirmación.

Se lanzará el mismo editor de confirmación, pero verás que ya incluye el mensaje de tu confirmación anterior. Puedes editar el mensaje como siempre y se sobreescribirá tu confirmación anterior

Deshacer una confirmacion y preparar nuevos archivos para la misma confirmacion

pg:45

Por ejemplo, si confirmas y luego te das cuenta que olvidaste preparar los cambios de un archivo que querías incluir en esta confirmación, puedes hacer lo siguiente:

  • git commit -m "description": Commit inicial en el cual olvide incluir un archivo.txt
  • git add archivo.tx: Agrego el archivo al area de preparacion (este archivo olvide icluir en la confirmacion anterior)
  • git commit --amend:Al final terminarás con una sola confirmación - la segunda confirmación reemplaza el resultado de la primera.Incluyendo en la confirmacion el archivo.tx.
